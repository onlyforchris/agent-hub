# 待办与通知系统设计

## 1. 概述

本系统包含两套关联能力：

- **待办系统**：从 Agent 对话中创建待办事项，支持截止日期、提醒时间、RBAC 可见范围控制。
- **站内通知系统**：支持按用户/角色/部门/全组织发送通知，已读未读追踪，消息模板机制。

两者集成点：待办创建时如有 `remindAt` → 写入提醒记录 → 定时任务扫描 → 通过模板生成站内通知。

---

## 2. 站内通知系统

### 2.1 核心模型

通知采用 **发送时展开** 模型：发送时立即将目标范围（USER/ROLE/DEPARTMENT/ORG）解析为具体收件人列表，每人一条 `agent_notification_receipt` 记录。

**为什么不用运行时解析？** 如果用户后来加入某个角色，不应看到加入前的历史通知。发送时展开语义正确，查询也简单高效（直接 `WHERE recipient_id = ?`）。

### 2.2 数据库表

#### agent_notification（通知主体）

| 列 | 类型 | 说明 |
|---|---|---|
| id | text PK | UUID |
| title | text NOT NULL | 通知标题 |
| content | text | 通知正文，支持 Markdown |
| category | text NOT NULL | SYSTEM / TODO / AGENT / ANNOUNCEMENT |
| sender_id | text | 发送者用户 ID，系统通知为 'system' |
| target_type | text NOT NULL | USER / ROLE / DEPARTMENT / ORG |
| target_id | text | 目标主体 ID，ORG 时为空 |
| template_id | text | 关联消息模板 ID |
| channel | text DEFAULT 'IN_APP' | 渠道：IN_APP / EMAIL / DINGTALK / WECOM / SMS |
| priority | int4 DEFAULT 0 | 优先级：0=普通, 1=重要, 2=紧急 |
| status | int4 DEFAULT 1 | 0=暂存, 1=正常, 2=删除 |

#### agent_notification_receipt（收件记录）

| 列 | 类型 | 说明 |
|---|---|---|
| id | text PK | UUID |
| notification_id | text NOT NULL | 关联通知 |
| recipient_id | text NOT NULL | 收件人用户 ID |
| is_read | int4 DEFAULT 0 | 0=未读, 1=已读 |
| read_at | timestamp(6) | 阅读时间 |
| is_deleted | int4 DEFAULT 0 | 0=正常, 1=用户已删除 |

#### agent_notification_template（消息模板）

| 列 | 类型 | 说明 |
|---|---|---|
| id | text PK | UUID |
| template_code | text UNIQUE | 模板编码，如 `todo.reminder` |
| title_template | text NOT NULL | 标题模板，支持 `{{变量}}` 占位 |
| content_template | text NOT NULL | 内容模板 |
| variables | JSONB | 变量定义 `[{"name":"todoTitle","required":true}]` |
| channel | text DEFAULT 'IN_APP' | 默认发送渠道 |

### 2.3 预置模板

| 编码 | 标题模板 | 用途 |
|---|---|---|
| `todo.reminder` | 待办提醒: {{todoTitle}} | 待办到期提醒 |
| `todo.assigned` | 新的待办: {{todoTitle}} | 待办分配通知 |
| `agent.notify` | Agent 通知: {{title}} | 通用 Agent 通知 |

### 2.4 RBAC 发送范围

| target_type | 解析方式 | 适用场景 |
|---|---|---|
| USER | 直接使用 target_id | 点对点通知 |
| ROLE | `selectUserIdsByRoleId(roleId)` | 通知某角色下所有用户 |
| DEPARTMENT | `selectUserIdsByDepartmentId(deptId)` | 通知某部门下所有用户 |
| ORG | `selectAllActiveUserIds()` | 全站公告 |

### 2.5 API

| 方法 | 路径 | 说明 | 权限码 |
|---|---|---|---|
| GET | `/api/notifications` | 当前用户通知列表（分页） | `notification:list` |
| GET | `/api/notifications/unread-count` | 未读数量 | `notification:count` |
| PUT | `/api/notifications/{id}/read` | 标记单条已读 | `notification:read` |
| PUT | `/api/notifications/read-all` | 全部标记已读 | `notification:read-all` |
| DELETE | `/api/notifications/{id}` | 删除通知（is_deleted=1） | `notification:delete` |
| POST | `/api/notifications` | 发送通知 | `notification:send` |

### 2.6 前端通知中心

- **通知铃铛**：侧边栏放置 `<NotificationBell />`，显示未读数量 Badge（最大 99+）
- **通知抽屉**：点击铃铛打开 Ant Design Drawer（右侧滑出，420px 宽）
- **通知列表**：每条显示类别图标、标题、内容摘要、时间；未读通知有蓝色圆点标识和浅灰背景
- **操作**：单条标记已读、单条删除、全部标记已读
- **轮询**：每 30 秒请求 `/api/notifications/unread-count` 更新徽标
- **已读/未读**：`is_read` 字段驱动，已读通知无圆点和背景色

## 3. 待办系统

### 3.1 数据库表

#### agent_todo（待办主体）

| 列 | 类型 | 说明 |
|---|---|---|
| id | text PK | UUID |
| title | text NOT NULL | 待办标题 |
| description | text | 详细描述 |

| created_by | text | 创建人用户 ID |
| status | int4 DEFAULT 0 | 0=待开始, 1=进行中, 2=已完成, 3=已取消, 99=删除 |
| priority | int4 DEFAULT 0 | 0=无, 1=低, 2=中, 3=高, 4=紧急 |
| due_date | timestamp(6) | 截止日期 |

| calendar_event_id | text | 关联日历事件 ID |
| visibility | text DEFAULT 'private' | private / team / org |

#### agent_todo_acl（行级权限）

| 列 | 类型 | 说明 |
|---|---|---|
| id | text PK | UUID |
| todo_id | text NOT NULL | 关联待办 |
| subject_type | text NOT NULL | USER / ROLE / DEPARTMENT |
| subject_id | text NOT NULL | 授权主体 ID |
| permission | text DEFAULT 'READ' | READ / WRITE |

#### agent_todo_reminder（提醒记录）

| 列 | 类型 | 说明 |
|---|---|---|
| id | text PK | UUID |
| todo_id | text NOT NULL | 关联待办 |
| channel | text DEFAULT 'IN_APP' | 提醒渠道 |
| scheduled_at | timestamp(6) NOT NULL | 计划发送时间 |
| sent_at | timestamp(6) | 实际发送时间 |
| status | int4 DEFAULT 0 | 0=待发送, 1=已发送, 2=失败, 3=已取消 |
| retry_count | int4 DEFAULT 0 | 重试次数（上限 4 次，指数退避） |

### 3.2 RBAC 行级权限

| visibility | 读权限 | 写权限 |
|---|---|---|
| private | 负责人 + 创建人 | 负责人 + 创建人 |
| team | 负责人 + 创建人 + ACL 匹配的 ROLE/DEPARTMENT | 负责人 + 创建人 + ACL 中 WRITE 权限主体 |
| org | 租户内全员 | 负责人 + 创建人 + 管理员 |

### 3.3 API

| 方法 | 路径 | 说明 | 权限码 |
|---|---|---|---|
| GET | `/api/todos` | 查询待办列表（RBAC 过滤） | `todo:list` |
| POST | `/api/todos` | 创建待办 | `todo:create` |
| GET | `/api/todos/{id}` | 查询单条待办 | `todo:view` |
| PUT | `/api/todos/{id}` | 更新待办 | `todo:edit` |
| DELETE | `/api/todos/{id}` | 删除待办（status=99） | `todo:delete` |

### 3.4 Agent Tool 集成

两个 Tool 由 `LifeAssistantAgent`（通用助手）通过 ReAct 循环调用：

- **todo.create**：必填 `title`，可选 `assignee`、`dueDate`、`remindAt`、`visibility`
  - `remindAt` / `dueDate` 支持绝对时间（yyyy-MM-dd HH:mm:ss）和相对时间（5分钟后/1小时后/明天/后天）
  - `assignee` 传 `current-user` 时自动解析为当前登录用户
  - 创建前 Agent 必须通过 `CONFIRM_NEEDED` 让用户确认内容
- **todo.list**：按 `assignee` + `status` 过滤（status: open/done/all），无需确认

Tool 内部已从 TodoStore（内存）切换为 TodoService（数据库），调用方无感知。

## 4. 提醒发送流程

采用 **TaskScheduler 精准调度 + 低频兜底轮询** 双层架构。

### 4.1 主路径：精准调度

```
todo.create (remindAt 非空)
  ├─ agent_todo_reminder 写入 (status=0, scheduled_at=remindAt)
  └─ TodoReminderScheduler.scheduleReminder()
       └─ TaskScheduler.schedule(task, remindAt)
            └─ 到点准时触发 fireReminder()
                 ├─ NotificationService.sendSystem() → agent_notification + receipt
                 └─ 更新 reminder: status=1, sent_at=now()
```

每条提醒一个独立定时任务，`TaskScheduler` 在精确时刻触发，延迟 < 1 秒。

### 4.2 兜底路径：低频轮询（每 5 分钟）

```
@Scheduled(fixedDelay = 300_000)
  → 扫描 agent_todo_reminder WHERE status=0 AND scheduled_at <= now()
  → 跳过已由 TaskScheduler 调度的（scheduledTasks.containsKey）
  → 处理因重启、GC 停顿等极端情况遗漏的提醒
```

### 4.3 启动恢复

```
@PostConstruct reloadPending()
  → 查询 status=0 AND scheduled_at > now() 的所有提醒
  → 逐条 scheduleReminder() 重新调度
```

### 4.4 失败重试

```
fireReminder() 失败
  → retry_count + 1
  → 指数退避: 1min / 5min / 15min / 60min
  → 更新 scheduled_at + re-scheduleReminder()
  → 超过 4 次 → status=2（失败）
```

### 4.5 降级策略

| 场景 | 处理 |
|---|---|
| 日历服务不可用 | Todo 创建成功，`calendarLinked=false`，后续异步补录 |
| 提醒写入失败 | 与 Todo 同事务回滚（同库同事务） |
| 通知发送失败 | 重试 4 次后退为失败状态，不阻塞 Todo 主流程 |
| `remind_at >= due_date` | 业务层拒绝，返回 `TODO003_INVALID_REMIND_TIME` |
| 服务器重启 | `@PostConstruct` 恢复所有未到期提醒 |
| TaskScheduler 漏调度 | 5 分钟一次兜底轮询补发 |

## 5. 错误码

| 错误码 | 说明 |
|---|---|
| `TODO001_TITLE_REQUIRED` | 待办标题不能为空 |
| `TODO002_TODO_NOT_FOUND` | 待办不存在 |
| `TODO003_INVALID_REMIND_TIME` | remind_at 必须早于 due_date |
| `TODO004_PERMISSION_DENIED` | 无权限操作该待办 |
| `N001_TITLE_REQUIRED` | 通知标题不能为空 |
| `N002_TEMPLATE_NOT_FOUND` | 消息模板不存在 |
| `N003_RECEIPT_NOT_FOUND` | 通知记录不存在 |
