# Agent 中台产品方案 v1.0

> 本文档供产品与研发对齐使用。描述 M1 阶段可交付的产品形态、平台边界、Agent 层级和演进路线。
>
> 前置阅读（可选）：`研发负责人评审意见与重设计方案.md`、`关于自定义创建Agent与DataAgent的技术评估.md`

---

## 1. 产品定位

### 一期（M1）

**AI 辅助的垂直业务 Agent 工具集**。首个交付物是「现金流预测 Agent」——财务人员通过对话式交互，从资金管理系统取数，完成 13 周滚动现金流预测，获得可解释的简报和可下载的预测报表。

### 长期（M2+）

在这套基座能力之上，扩展更多垂直 Agent（外汇敞口、智能排程、偏差归因等），并在后续版本开放查询型 Agent 的配置化创建。

**一期明确不是：**
- Agent 操作系统或流程编排平台
- 数字人工厂或 Agent 市场
- 可视化工作流编排器（Skill Studio）
- 多 Agent 自主协同系统

### 3.2 核心接口

```java
// 每个垂直Agent继承这个
public abstract class AgentBase {

    // 子类必须实现
    abstract AgentInfo info();
    abstract Intent classify(String input, SessionContext ctx);
    abstract AgentResult doExecute(Intent intent, SessionContext ctx);
    abstract List<ToolHandler> tools();

    // 可选覆盖
    String routeHint() { return info().description(); }
    String model() { return "deepseek-v3"; }

    // 平台提供的能力(final, 不可覆写)
    final AgentResult execute(Intent intent, SessionContext ctx) {
        // 统一入口: trace初始化 → doExecute → 异常兜底 → audit → 返回
    }

    // 子类调用的平台能力
    protected ToolResult callTool(String key, Map<String,Object> params);
    protected LlmResult callLlm(String promptKey, Map<String,Object> vars);
    protected void audit(String action, Map<String,Object> detail);
}
```

### 3.3 请求链路

```
POST /api/agent/ask { input, agentId?, sessionId }
        │
        ▼
┌──────────────────┐
│ AgentController   │
└──────┬───────────┘
       │
       ├─ agentId 指定 ──→ 直达指定Agent
       │
       └─ agentId 为空 ──→ AgentRouter
              │
              ├─ Gatekeeper (1次LLM)
              │    └─ type=chat → 纯对话, 不调Agent
              │    └─ type=task → 继续
              │
              ├─ AgentRouter.LLM routing (1次LLM)
              │    └─ 匹配Agent → 对应Agent
              │    └─ 匹配失败 → 引导用户选择Agent
              │
              ▼
       AgentBase.execute()
              │
              ├─ classify(input) → Intent
              ├─ doExecute(intent) → AgentResult
              │    ├─ callTool() → ToolExecutor → TMS等
              │    ├─ callLlm() → LlmGateway → DeepSeek等
              │    └─ audit() → AuditService → DB
              │
              ▼
       AgentResult { reply, data, actions, trace }
```

## 4. Agent 的三层创建体系

| 层级 | 能力 | 一期 | 二期 |
|------|------|:----:|:----:|
| **L1 参数级** | 修改已有 Agent 的名称/描述/权限/用户分配 | ✅ | 增强 |
| **L2 查询型** | 选 Tool + 写 Prompt = 查询型 Agent（单步调用） | — | ✅ |
| **L3 流程型** | 可视化编排多步骤 + 条件分支 + 业务规则 | 永不 | 永不 |

### L1 详情（一期交付）

管理后台中，管理员可对已有 Agent 进行：
- 修改名称、描述、图标（纯展示，不影响逻辑）
- 调整权限级别
- 分配可见范围（哪些用户/角色可用）
- 启用/禁用

Java Agent 的执行逻辑不开放修改。

### L2 详情（二期规划）

管理后台中，管理员可创建「查询型 Agent」：
- 从 Tool 目录中选择 1-5 个 Tool
- 编写 system prompt
- 配置使用的 LLM 模型
- 设置权限级别

创建后的 Agent 自动出现在工作台的 Agent 标签中。执行逻辑为通用模板：「LLM 理解意图 → 匹配最合适的 Tool → 调用 → 润色回复」。

**硬约束：不支持多步骤串联、不支持条件分支、不支持自定义校验规则。**

## 5. 首个 Agent：现金流预测

### 5.1 核心流程

```
用户输入 → classify (LLM意图识别)
    │
    ├─ intent=forecast → {
    │     1. 调 ledger.fetch 从TMS取分类账
    │     2. 校验数据完整性 (非空、字段齐全、日期连续)
    │     3. ForecastEngine 计算13周滚动预测
    │     4. LLM 生成简报 (基于结构化预测结果)
    │     5. 审计落库
    │  } → AgentResult { reply, 预测数据, 图表配置, Excel }
    │
    ├─ intent=explain → { 对已有预测结果做解释和追问 }
    ├─ intent=export  → { 导出Excel/PDF报表 }
    └─ intent=chat    → { 纯对话,不调Tool }
```

### 5.2 数据源

**一期：** 直连内部资金管理系统（TMS）数据库，只读账号，视图查询。

**数据范围：**
- 银行账户（账户编码、行名、币种、余额）
- 资金流水（日期、摘要、借方/贷方金额、余额、对方科目）
- 应收应付（到期日、金额、客商、合同号）

**连接方式：** 部署时配置数据源连接信息（URL/账号/密码），密码通过环境变量注入。

**安全约束：** Agent 只有只读权限，不写、不改、不删。

### 5.3 预测算法

```text
ForecastEngine 职责:
  1. 按科目分组 (经营流入/流出、融资流入/流出、投资流入/流出)
  2. 计算历史每周各类别的日均值
  3. 识别周期性模式 (月度、季度、年度)
  4. 考虑已知的未来事件 (大额合同到期、税款缴纳日等) — 二期
  5. 生成13周滚动预测 (每周的流入/流出/净额/期末余额)
  6. 标注缺口周（期末余额为负的周）
  7. 给出置信度
```

**一期算法：** 移动平均 + 周期性因子。简单但口径明确、可解释。

### 5.4 前端页面

Agent 对话 + 预测看板两个页面：

**对话页：** 通用工作台，Agent 标签 + 对话列表 + 输出区（含简报卡片和快捷操作按钮）

**看板页：** 预测趋势图 + 明细表 + 简报文字块 + 导出按钮

## 8. MVP 验收标准

### 业务验收
- [ ] 财务人员能通过对话独立完成一次 13 周现金流预测
- [ ] 预测结果能逐项追溯到 TMS 数据来源和计算口径
- [ ] 导出的 Excel 报表可直接用于现有工作流
- [ ] 用户能基于预测结果继续追问或调整参数

### 技术验收
- [ ] 同一输入数据和参数下，预测数值结果稳定一致
- [ ] LLM 生成的简报内容必须引用结构化预测结果，不可凭空编造数据
- [ ] 每次执行有完整 trace/audit 记录，可定位数据来源、参数、用户
- [ ] 权限不足的用户看不到 Agent 或被拒绝执行
- [ ] TMS 数据库连接的密码不出现于配置文件或日志中
