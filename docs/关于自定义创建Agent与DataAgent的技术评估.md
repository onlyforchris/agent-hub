# 关于自定义创建 Agent 与 DataAgent 的技术评估

> 评估背景：产品提出允许管理员/用户通过"勾选 Tool + 配置 Prompt + 关联 Skill"的方式快速创建 Agent（类似数字人/DataAgent 概念）。
> 本文档从工程角度分析该方案的可行边界，并给出分阶段建议。

---

## 1. 先区分两种 Agent

### 查询型 Agent（单步 · LLM路由 + 一个Tool + 润色回复）

**特征：** 用户意图 → 选一个Tool → 调Tool拿结果 → LLM翻译成自然语言

**典型场景：**

| Agent | 用户输入 | 调用的 Tool | LLM 做什么 |
|-------|---------|------------|-----------|
| 头寸查询 | "今天银行账户余额多少" | cash.position | 把数值表翻译成一段话 |
| 汇率助手 | "美元汇率最近怎么样" | fx.rate | 解释汇率走势 |
| 日报导出 | "帮我导出华南区上周资金日报" | delivery.excel_daily | 确认导出结果 |

**这类 Agent 的逻辑是通用的，可以通过配置生成。**

---

### 流程型 Agent（多步 · 领域逻辑 · 计算 · 校验 · 异常处理）

**特征：** 多步骤串联，每步有业务判断，中间结果不是简单的 Tool 返回值透传

**典型场景：**

| Agent | 步骤 | 为什么不能配置化 |
|-------|------|-----------------|
| 现金流预测 | 取分类账 → 按科目分组 → 校验数据完整性 → 计算历史均值 → 识别周期性 → 生成13周滚动预测 → 标注缺口 → 生成简报 | 分组逻辑、预测算法、异常判断——这些是财务领域的代码，不是"调哪个Tool"的问题 |
| 外汇敞口 | 取头寸 → 按币种汇总 → 关联汇率 → 计算敞口 → 压力测试 → 风险评级 | 敞口计算有会计口径，汇率关联有特定规则，压力测试有参数——无法暴露成配置表单 |
| 智能排程 | 取应收应付 → 匹配规则 → 生成排程方案 → 优化 → 冲突检测 | 排程是一个 NP-hard 优化问题，不是调用链问题 |

---

## 2. 为什么"勾选 Tool + 写 Prompt"不能做出流程型 Agent

### 2.1 调用顺序不是 prompt 能保证的

给 LLM 3 个 Tool，让它自主决定调用顺序：

- 用户说"做现金流预测"，它应该先调 ledger.fetch 还是 compute.stats？参数怎么填？
- 如果 ledger.fetch 返回 500 行数据，LLM 会不会自作主张截断？会不会漏字段？会不会幻觉？
- 同一个用户输入，两次调用的 Tool 序列可能不同——审计怎么办？

**确定性的调用顺序是业务要求，但 LLM 不保证确定性。**

### 2.2 中间数据处理不是 Tool 的 inputSchema 能覆盖的

Tool A 的输出 → Tool B 的输入，中间可能需要：

- 字段映射（TMS 的 `debit_amount` → 预测引擎的 `inflow`）
- 数据清洗（去重、补缺失值、滤除已冲销条目）
- 口径转换（本位币 vs 原币、含税 vs 不含税）
- 分组聚合（按科目、按账户、按成本中心）

**这些不是配置，是代码。** Tool 的 inputSchema 描述的是参数格式，不是数据处理逻辑。

### 2.3 异常判断需要业务知识

- 分类账数据不全 → 是中止执行还是降级处理？降级到什么程度？
- 预测结果出现极端值 → 是数据错了、算法错了、还是真的发生了极端事件？
- 用户要调一个无权使用的 Tool → 拒绝、申请临时授权、还是换一个替代方案？

**这些判断需要理解业务上下文。Prompt 里写不完所有异常情况。**

### 2.4 市场上没有人真正做到

Dify 的工作流编辑器允许可视化工编排，看起来像"业务人员也能配"。但实际使用的团队里，**能独立配置工作流的都是技术人员**。非技术人员能做到的是：修改已有流程里的文案参数、开关某个步骤——这正是 L1 的能力范围。

OpenAI GPTs（自定义 GPT）允许"选工具 + 写指令"创建一个定制 ChatGPT。但 OpenAI 公开的数据显示，绝大多数 GPTs 的使用场景是"角色设定 + 知识库注入"，极少有 GPTs 能可靠执行多步骤工具调用链。

## 3. 可行方案：三层的 Agent 创建体系

### L1：参数级（一期做）

**能力：** 对已有 Agent 修改名称、描述、头像、权限、启用/禁用、分配用户

**本质：** Agent 的 Java 逻辑不变，只改管理属性

**适用：** "给审计团队复制一个现金流Agent，设成只读"
"把外汇Agent的权限从 LEVEL_3 降到 LEVEL_2"
"华南区只用头寸查询，隐藏预测功能"

**实现：** 管理后台表单，AgentConfig 持久化，覆盖 AgentBase 的 info()

---

### L2：查询型 Agent 配置（一期后半段或二期做）

**能力：** 选一个或多个 Tool + 写 system prompt + 配置 LLM 模型 = 新的查询型 Agent

**核心约束：** **只能做单步骤调用。** 执行逻辑是固定模板——"LLM 理解意图 → 选一个最匹配的 Tool → 调 Tool → LLM 解释结果"

**适用：** "头寸查询数字人"、"日报导出助手"、"汇率播报员"

**不适用的明确红线：** 凡是需要"先调A再调B""计算结果""做规则判断"的，一律不适用，必须写 Java Agent

**实现：** `ConfiguredAgent extends AgentBase`，`doExecute()` 内为通用模板（不开放自定义）

---

### L3：流程级（永远不做）

**能力：** 可视化编排步骤、写条件分支、自定义业务规则

**为什么永远不做：**

1. 这是 Skill Studio 的翻版，会把团队拖入工作流引擎的维护黑洞
2. 市场上有 Dify/LangGraph/Temporal 在做这件事，且投入远超你的团队规模
3. 财务领域的编排复杂度远高于通用场景——需要特定的会计知识才能排对流程
4. 即使做出来了，能使用它的人（懂财务+懂技术）极少，ROI 为负

---

## 4. 建议分阶段策略

| 阶段 | 内容 | 面向用户 | 交付形态 |
|------|------|---------|---------|
| **M1 当前** | Java 手写的 CashflowAgent（现金流预测） | 内部验证 / 售前演示 | 一个完整可用的 Agent |
| **M2** | L1 管理能力：Agent 基本信息编辑、权限分配、启用/禁用 | 实施人员 / 管理员 | 管理后台页面 |
| **M3** | 第 2-3 个 Java Agent（外汇敞口 / 智能排程） | 业务用户 | 新增 Agent |
| **M4** | L2 查询型配置：选 Tool + 写 Prompt = 新 Agent | 实施人员 | Agent 创建表单 |

## 5. 跟产品的沟通要点

### 能做的

> "产品内部把 Agent 分成两类：一类是复杂流程型（如现金流预测），这类由研发团队用代码实现，保证确定性和可靠性；另一类是查询型（如头寸查询、汇率播报），这类可以通过管理后台配置快速生成。我们在一期先交付复杂流程型 Agent（现金流预测），查询型配置能力在后续版本中提供。"

### 做不了的（以及为什么）

> "市场上没有人做到'勾选 Tool + 写 Prompt 就能创建复杂业务 Agent'。Dify、GPTs、Coze 都做不到。原因不是技术不够强，而是**多步骤业务逻辑 = 代码**，无法压缩进配置表单。如果我们在产品中承诺这个能力，最终交付的会是一个体验极差的半成品——用户配了一个'现金流预测Agent'，但输出的预测结果不稳定、不可信、不可审计。"

### 建议产品侧同步做的事

> "我们建议产品团队同步做一件事：找 2-3 个真实的财务用户，观察他们实际的工作流程。我们需要理解：他们是在一个系统里完成全流程？还是在多个系统间跳转？他们最耗时的环节是什么？他们的输出物是什么格式？这样我们能确保研发的不是'我们觉得有用的Agent'，而是'他们愿意掏钱的工具'。"
