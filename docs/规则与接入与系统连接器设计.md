# 规则与接入与系统连接器设计

## 1. 概述

为对齐产品原型中的 **「规则与接入」** 能力分组，工作台侧栏将原顶层 **「消息中心」** 收纳为该分组下的子菜单，并新增同级子菜单 **「系统连接器」**：用于登记外部 **MySQL** 数据源连接（加密存储凭据）、校验连通性、拉取 **库表元数据**，为后续 **Text2SQL**（自然语言 → 受控 SQL → 只读查询）提供可审计的数据访问基础。

本设计遵循 v2.0 约定：**业务决策与高风险操作不在 LLM 内完成**；连接器仅提供「连接与元数据」能力，不在本期开放任意 SQL 执行入口。

---

## 2. 信息架构与前端路由

### 2.1 侧栏结构

| 层级 | 名称 | 路由 | 前端组件 |
|------|------|------|-----------|
| 一级 | Agent 工作台 | `/` | `AgentWorkspacePage` |
| 一级 | 预测看板 | `/forecast` | `ForecastPage` |
| 一级 | **规则与接入** | （目录，无独立路由） | 无 `component`，`path` 为空 |
| 二级 | 消息中心 | `/notifications` | `NotificationCenterPage` |
| 二级 | 系统连接器 | `/connectors` | `SystemConnectorPage` |
| 一级 | 管理后台 | `/settings` | `SettingsPage` |

侧栏使用 Ant Design `Menu` 的 `items` 子菜单；进入 `/notifications` 或 `/connectors` 时自动展开「规则与接入」分组。

### 2.2 与 RBAC 菜单表的关系

`sys_menu` 中需存在与上表一致的记录（含 `parent_id` 树形关系），以便 **管理后台菜单管理**、**按角色授权菜单** 与 **用户路由接口** 语义一致。工作台当前仍以 **前端编排侧栏** 为主；若后续改为完全由 `/api/rbac/menus/routes` 驱动，应保持同一套 `id / path / component`。

## 3. 数据库设计

### 3.1 菜单与资源归属调整

- **规则与接入**（建议 `id = menu-rules-access`）：`parent_id = '0'`，`path`、`component` 为空，仅作目录。
- **消息中心**（`menu-message-center`）：`parent_id = menu-rules-access`，`path = '/notifications'`。
- **系统连接器**（`menu-system-connector`）：`parent_id = menu-rules-access`，`path = '/connectors'`。

站内通知相关 **API 资源**（`notification:list` 等）的 `sys_resource.menu_id` 从工作台菜单改为 **消息中心菜单**，便于在「资源 ↔ 菜单」维度做审计与配置管理。

> 说明：`application.yml` 中 `/api/notifications/**` 仍可能配置在 **authenticated-paths**（登录即可调接口）。这与 **资源表中的 menu_id 归属** 是两件独立事情：前者影响网关是否做 `resource_code` 校验；后者影响 RBAC 元数据与后台展示。

### 3.2 表：`text2sql_data_connection`

定义见 `database/init/001_schema.sql`（与 `003_menu_rules_connector.sql` 中幂等建表段一致）。

| 列 | 说明 |
|----|------|
| id | 主键，应用层 UUID |
| display_name | 连接显示名 |
| db_type | 一期仅 `MYSQL` |
| host / port / database_name / username | 连接参数；库名即 MySQL schema |
| password_enc | **AES 密文**，不落明文 |
| jdbc_params | 可选附加 JDBC 查询串片段（长度与字符集受限，见实现） |
| status / remark / create_by / create_time / update_by / update_time | 公共字段；删除为 `status = 2` |

## 4. API 与权限码

### 4.1 设计原则

- `sys_resource.resource_code` **全局唯一**。同一 `INSERT ... ON CONFLICT (resource_code) DO UPDATE` 批次内 **禁止** 两条记录使用相同 `resource_code`（否则会触发 PostgreSQL 错误：`cannot affect row a second time`）。
- 列表与详情两个 **GET** 分别使用：
  - `connector:datasource:list` — `GET /api/text2sql/connectors`
  - `connector:datasource:detail` — `GET /api/text2sql/connectors/{id}`

### 4.2 资源一览

| resource_code | Method | Path | 说明 |
|----------------|--------|------|------|
| connector:datasource:list | GET | `/api/text2sql/connectors` | 列表 |
| connector:datasource:detail | GET | `/api/text2sql/connectors/{id}` | 详情 |
| connector:datasource:add | POST | `/api/text2sql/connectors` | 新建 |
| connector:datasource:edit | PUT | `/api/text2sql/connectors/{id}` | 更新 |
| connector:datasource:delete | DELETE | `/api/text2sql/connectors/{id}` | 逻辑删除 |
| connector:datasource:test | POST | `/api/text2sql/connectors/test` | 按表单参数探测连接 |
| connector:datasource:test-saved | POST | `/api/text2sql/connectors/{id}/test` | 使用已存密文探测 |
| connector:datasource:metadata | GET | `/api/text2sql/connectors/{id}/metadata` | 拉取表清单（`information_schema`） |

上述资源默认挂在 `menu-system-connector` 上；**未**放入 `authenticated-paths` 时，需经 `ResourceAuthorizationFilter` 做 **resource_code** 校验（超级管理员跳过）。

### 4.3 角色授权建议

- **超级管理员**：登录时下发的 `permissions` / `menuIds` / `agentIds` 与网关策略一致，为 **`sys_resource` / `sys_menu` / `agent` 表中的全量有效数据**（无需依赖每次新增资源后手工补 `sys_role_resource`）。
- **司库操作员等业务角色**：在 `sys_role_menu` 中授予 `menu-rules-access` 及其子菜单（否则树形菜单可能缺少父节点）；在 `sys_role_resource` 中按需授予上表权限码（种子中已包含连接器全套权限作为示例，可按最小权限收紧）。

### 4.4 菜单 `permission_code`

`menu-system-connector` 的 `permission_code` 与入口语义对齐，使用 **`connector:datasource:list`**（与列表资源码一致，便于「有列表权即可进页」；详情由独立 `connector:datasource:detail` 控制单条读取）。

### 4.5 超级管理员与登录下发的 `permissions`

历史上 `login` / `refresh` 返回的 `permissions` 仅来自 `sys_role_resource` 按角色展开；**新增 `sys_resource` 行后**，若未同步插入 `sys_role_resource`，或用户仍持有旧 JWT，则前端 `usePermission` 会为假。

与 `ResourceAuthorizationFilter`（超级管理员跳过资源校验）对齐：**凡拥有 `role_type = SUPER_ADMIN` 角色的用户**，`RbacService.permissionsByUser` 改为下发 **`sys_resource` 中全部 `status = 1` 的 `resource_code`**；`menuIdsByUser`、`agentIdsByUser` 同理下发全量菜单 ID 与 Agent ID，避免管理端与前端 UX 依赖「手工补全 role_resource 行」。

变更后需 **重新登录或刷新 token** 以拉取新列表；若使用 Redis 缓存 RBAC，需等待 TTL 失效或触发既有 `evictUserCaches` 逻辑。

## 5. 后端实现要点

### 5.1 模块位置

- 实体与 Mapper：`agent-server-system`，表名 `text2sql_data_connection`。
- 服务与控制器：`com.efloow.agenthub.system.text2sql` 包（如 `Text2SqlConnectionService`、`Text2SqlConnectionController`）。
- MySQL 驱动：`mysql-connector-j`（`runtime`），通过 `DriverManager` 短连接做探测与元数据读取，设置 **登录超时**，避免悬挂。

### 5.2 密码与密钥

- 复用配置项 **`agent.llm.encryption-secret`**（或等价环境变量）与公共 **`AesUtil`** 加解密，避免再引入一套密钥管理。
- 日志与接口响应中 **禁止** 输出明文密码或完整 JDBC URL 中的敏感片段。

### 5.3 参数校验

- 对 `host`、`database_name`、`jdbc_params` 等参与 URL 拼接的字段做 **白名单/长度** 约束，降低 JDBC URL 注入面（实现以代码为准）。

### 5.4 错误码（示例）

| 码 | 含义 |
|----|------|
| C002_INVALID_PARAM | 参数非法（含校验失败） |
| C003_UNSUPPORTED_DB | 不支持的数据库类型 |
| D001_DB_CONNECT_FAILED | 连接失败 |
| D002_DB_METADATA_FAILED | 元数据读取失败 |
| D003_CONNECTION_NOT_FOUND | 连接不存在或已删除 |

## 6. 前端实现要点

- 路由：`/connectors` → `SystemConnectorPage`。
- 页面入口可见性：`usePermission(['connector:datasource:list', 'connector:datasource:detail'])` 二选一即可进入列表能力（与后端列表/详情接口一致）。
- 操作按钮：`Permission` 组件绑定 `connector:datasource:add|edit|delete|test|metadata` 等。
- HTTP 封装：`agent-web/src/api/text2sqlConnectorApi.ts`，统一走 `httpClient`。

| 文件 | 用途 |
|------|------|
| `database/init/002_seed.sql` | 新环境：菜单、资源、`sys_role_menu` / `sys_role_resource` 完整种子 |

`003` 末尾包含对历史错误 **`resource_code` 重复** 的纠偏 `UPDATE`（若曾将两条 GET 误写为同一 `connector:datasource:view`），执行顺序与幂等性以脚本内注释为准。

**上线操作建议**：在事务中执行 `003`（若你的发布流程支持），执行后通知用户 **重新登录** 以刷新 JWT 附带权限列表。

## 8. 安全与合规

- 连接器仅用于 **只读分析链路** 的铺垫；生产若开放 SQL 执行，必须单独设计：**白名单表、行级权限、SQL 语法限制、结果集上限、独立审计与 traceId** 等，不在本文档一期范围内。
- 外部 MySQL 与平台 PostgreSQL **物理隔离**；凭据仅存于本平台库密文字段。
- 满足项目通用要求：RBAC 资源登记、`unregistered-resource-policy = DENY`、敏感信息不进日志。

---

## 9. 后续演进（建议 backlog）

1. **Text2SQL 执行层**：在 LlmGateway 之外增加「SQL 生成 → 静态校验 → 参数化执行 → 结果集脱敏」流水线；禁止 LLM 直接触库。
2. **多数据源类型**：PostgreSQL、ClickHouse 等，抽象 `db_type` 与连接工厂。
3. **连接级权限**：按部门/用户绑定可用 `connection_id`，与 `data_scope` 结合。
4. **连接池与限流**：高频查询场景下再引入 Hikari 分池与租户级配额。
5. **侧栏完全后端化**：由 `menus/routes` 返回树，减少前后端双份菜单维护。
