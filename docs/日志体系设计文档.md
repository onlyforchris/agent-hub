# Agent 中台日志体系设计文档

> 版本: v2.0 | 更新: 2026-05-02 | 适用范围: 全项目前后端

---

## 1. 设计目标

1. **全链路可控可查**：Controller → Router → Agent → Tool/LLM → Response 每个节点有日志
2. **traceId 贯通全流程**：统一使用 MDC traceId，不重复生成，一条请求链路一个 traceId
3. **按模块分层输出**：app / agent / tool / llm / audit / error 六个日志文件
4. **自动轮转**：单文件 50MB 上限，最多保留 30 个，归档 gz 压缩，硬盘上限 3GB

---

## 2. 日志文件分层

| 文件名 | 路径 | 级别 | 包路径 | 内容 |
|--------|------|------|--------|------|
| `logs/app.log` | 应用级 | INFO | `com.efloow.agenthub` (root) | 应用启动、配置加载、通用业务事件 |
| `logs/agent.log` | 编排层 | INFO | `base`, `controller`, `router`, `agent` | 路由决策、意图分类、Agent 执行/结果 |
| `logs/tool.log` | 工具层 | INFO | `application.tool`, `infrastructure.tool` | Tool 调用、参数、结果、耗时 |
| `logs/llm.log` | 模型层 | DEBUG | `application.llm`, `infrastructure.llm` | LLM 请求/响应摘要、token 用量、耗时 |
| `logs/audit.log` | 审计层 | INFO | `application.audit`, `system` | Trace 记录、访问审计、权限变更 |
| `logs/error.log` | 错误汇总 | ERROR | 全包 | 所有 ERROR 级别日志汇总 |

### 轮转策略

```
maxFileSize: 50MB       # 单文件最大 50MB
maxHistory: 30          # 最多保留 30 个归档文件
totalSizeCap: 500MB     # 单类日志硬盘上限 500MB (6 类总计 3GB)
压缩: gzip              # 归档文件压缩存储
```

## 3. 日志格式

```
%d{yyyy-MM-dd HH:mm:ss.SSS} %-5level [%X{traceId}] [%X{userId:-}] [%X{agentId:-}] [%t] %logger{36} - %msg%n
```

### 格式字段说明

| 字段 | 来源 | 说明 |
|------|------|------|
| `%d{yyyy-MM-dd HH:mm:ss.SSS}` | 系统 | 时间戳，精确到毫秒 |
| `%-5level` | SLF4J | 日志级别，左对齐 5 字符 |
| `[%X{traceId}]` | MDC | 全链路追踪 ID，由 TraceIdFilter 注入 |
| `[%X{userId:-}]` | MDC | 当前用户，未登录显示 `-` |
| `[%X{agentId:-}]` | MDC | 当前执行的 Agent，非 Agent 请求显示 `-` |
| `[%t]` | 系统 | 线程名 |
| `%logger{36}` | SLF4J | Logger 名，最大 36 字符 |
| `%msg%n` | SLF4J | 日志消息 + 换行 |

### 示例输出

```
2026-05-02 14:30:01.234 INFO  [a1b2c3d4] [user-001] [cashflow] [http-nio-8066-exec-1] c.e.a.r.AgentRouter - routing: mode=llm, agentId=cashflow, intent=forecast
2026-05-02 14:30:01.456 INFO  [a1b2c3d4] [user-001] [cashflow] [http-nio-8066-exec-1] c.e.a.b.AgentBase - agent execute start: agentId=cashflow, intent=forecast, userId=user-001, sessionId=sess-789
2026-05-02 14:30:01.789 INFO  [a1b2c3d4] [user-001] [cashflow] [http-nio-8066-exec-1] c.e.a.a.t.ToolExecutor - tool invoke done: toolKey=ledger.fetch, success=true, durationMs=234
2026-05-02 14:30:02.012 INFO  [a1b2c3d4] [user-001] [cashflow] [http-nio-8066-exec-1] c.e.a.i.l.DeepSeekLlmGateway - llm chat response: model=deepseek-v3, inputTokens=456, outputTokens=128, durationMs=1200
2026-05-02 14:30:02.234 INFO  [a1b2c3d4] [user-001] [cashflow] [http-nio-8066-exec-1] c.e.a.b.AgentBase - agent execute done: agentId=cashflow, intent=forecast, durationMs=2000
```

## 4. MDC 上下文管理

### MDC 字段生命周期

| MDC Key | 设置位置 | 设置时机 | 清除时机 |
|---------|----------|----------|----------|
| `traceId` | `TraceIdFilter` | 请求进入 | `TraceIdFilter.finally` |
| `userId` | `TraceIdFilter` | 请求进入（从 SecurityContext 读取） | `TraceIdFilter.finally` |
| `sessionId` | `AgentController` | Agent ask 请求 | 请求结束 |
| `agentId` | `AgentBase.execute()` | Agent 执行开始 | `AgentBase.execute() finally` |

### MDC 使用规则

1. `traceId` 优先从请求头 `X-Trace-Id` 获取，若无则生成 UUID
2. 响应头必须回传 `X-Trace-Id`
3. Agent 内部不再自行生成 traceId，统一使用 MDC 中的
4. `AgentBase.execute()` 若发现 MDC 无 traceId（非 HTTP 场景），自动生成
5. MDC 必须在 finally 中清理，禁止残留到下一个请求

## 5. 日志级别使用规范

| 级别 | 使用场景 | 示例 |
|------|----------|------|
| **ERROR** | 需要立即处理的问题：LLM 调用失败、数据库连接异常、外部系统不可用 | `log.error("agent execute failed: agentId={}, intent={}", agentId, intent, e)` |
| **WARN** | 可恢复异常：权限不足、Tool 未找到、参数校验失败、LLM fallback 降级 | `log.warn("tool permission denied: toolKey={}", toolKey)` |
| **INFO** | 关键业务节点：路由决策、Agent 执行开始/结束、Tool 调用、LLM 请求/响应、审计记录 | `log.info("agent execute done: agentId={}, intent={}, durationMs={}", agentId, intent, durationMs)` |
| **DEBUG** | 调试信息：LLM 请求详情、Gatekeeper 分类细节、Agent 匹配过程 | `log.debug("gatekeeper: type=TASK, input={}", userInput)` |

### 环境级别配置

```yaml
# 开发环境
logging.level.com.efloow.agenthub: DEBUG

# 生产环境
logging.level.com.efloow.agenthub: INFO
logging.level.com.efloow.agenthub.application.llm: INFO  # 生产关闭 LLM DEBUG
```

## 6. 全链路日志节点

```
请求进入
  │  TraceIdFilter          → [traceId] [userId] 注入 MDC
  │  AuditAccessLogService   → audit.log: 访问审计
  ▼
AgentController
  │  [sessionId] 注入 MDC
  │  agent.log: "agent ask: input=..., agentId=..., sessionId=..."
  ▼
AgentRouter
  │  agent.log: "routing: mode=llm/direct/chat/no_match, agentId=..., intent=..."
  ▼
Gatekeeper
  │  (DEBUG) gatekeeper 分类日志
  ▼
AgentBase.execute()
  │  [agentId] 注入 MDC
  │  agent.log: "agent execute start: agentId=..., intent=..."
  │
  ├── callTool()
  │     agent.log: "tool call start: toolKey=..."
  │     tool.log:  "tool invoke done: toolKey=..., success=..., durationMs=..."
  │     agent.log: "tool call done: toolKey=..., durationMs=..."
  │
  ├── callLlm()
  │     agent.log: "llm call start: model=..., promptKey=..."
  │     llm.log:   "llm chat request: provider=..., model=..., messageCount=..."
  │     llm.log:   "llm chat response: model=..., inputTokens=..., outputTokens=..., durationMs=..."
  │     agent.log: "llm call done: model=..., durationMs=..."
  │
  └── agent.log: "agent execute done: agentId=..., intent=..., durationMs=..."
  ▼
GlobalExceptionHandler (异常时)
  │  error.log: "unexpected error: traceId=..., uri=..."
  ▼
响应返回
  │  TraceIdFilter.finally → MDC 清理
```

## 7. 日志安全底线

绝对禁止在日志中输出：
1. API Key / Secret
2. 用户密码 / BCrypt hash
3. 完整银行账号 / 身份证号
4. 完整手机号 / 邮箱
5. 完整 Prompt 原文（生产环境）
6. JWT Token 原文

## 8. 日志查询指南

### 按 traceId 定位全链路

```bash
# 查找某个 traceId 的所有日志（按时间排序）
grep "a1b2c3d4" logs/*.log | sort

# 只看 agent 链路
grep "a1b2c3d4" logs/agent.log

# 只看错误
grep "a1b2c3d4" logs/error.log
```

### 按模块查看

```bash
# 查看最近的 Agent 执行记录
tail -100 logs/agent.log

# 查看 Tool 调用失败
grep "fail" logs/tool.log

# 查看 LLM 调用耗时
grep "durationMs" logs/llm.log

# 查看审计记录
grep "audit" logs/audit.log
```

### 按时间范围

```bash
# 归档文件已按日期命名，可直接定位
ls logs/agent.2026-05-02.*.log.gz

# 解压查看
gunzip -c logs/agent.2026-05-02.0.log.gz | grep "cashflow"
```

## 9. 关键类 Logger 清单

| 类 | Logger 名 | 日志文件 | 说明 |
|----|-----------|----------|------|
| `TraceIdFilter` | — | — | 仅设 MDC，不写日志 |
| `AgentController` | `AgentController.class` | agent.log | 请求入口日志 |
| `AgentRouter` | `AgentRouter.class` | agent.log | 路由决策日志 |
| `Gatekeeper` | `Gatekeeper.class` | agent.log | 门禁分类日志 |

| `DeepSeekLlmGateway` | `DeepSeekLlmGateway.class` | llm.log | LLM 请求/响应日志 |

| `AuditAccessLogService` | `AuditAccessLogService.class` | audit.log | 访问审计日志 |
| `GlobalExceptionHandler` | `GlobalExceptionHandler.class` | error.log | 异常日志 |

## 10. Logback 配置文件结构

```
logback-spring.xml
├── springProperty (appName)
├── property (LOG_PATTERN, LOG_DIR)
├── CONSOLE appender
├── APP_FILE (logs/app.log)       — RollingFileAppender
├── AGENT_FILE (logs/agent.log)   — RollingFileAppender
├── TOOL_FILE (logs/tool.log)     — RollingFileAppender
├── LLM_FILE (logs/llm.log)       — RollingFileAppender
├── AUDIT_FILE (logs/audit.log)   — RollingFileAppender
├── ERROR_FILE (logs/error.log)   — RollingFileAppender + LevelFilter(ERROR only)
├── 第三方框架 logger (WARN)
├── 业务模块 logger (INFO/DEBUG → 对应 appender, additivity=false)
└── root logger (INFO → CONSOLE + APP_FILE + ERROR_FILE)
```
