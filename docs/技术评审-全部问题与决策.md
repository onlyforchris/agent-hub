> 整理自研发负责人评审讨论。按问题域分类，每个条目包含问题描述和讨论结论。

## 一、自研 vs 开源

### 1. 为什么必须自研 Skill Runtime？开源方案具体卡在哪里？

**问题：** 当前方案自研了一套 Skill YAML 编排引擎（解析 YAML → 执行 Step → 变量传递 → 调 Tool/Rule/LLM → 记录 Trace）。LangGraph、Dify、OpenAI Agents SDK、Temporal 都有成熟方案。文档说 Dify "企业核心链路容易被平台抽象约束"——具体哪个约束？哪个接口？没有实际证据。

**结论：** 编排层不构成我们的差异化壁垒。应复用成熟框架，自研只保留：财务领域 Tool、预测和归因算法、数据连接器、报表模板、权限审计适配、私有化部署适配。

但考虑到团队是 Java 栈且只有 2 个后端，引入 Python 编排服务（LangGraph/Dify）会增加运维成本。取舍方案：

- **当前最优：** 保留 Java 基座，但砍掉 YAML DSL，Agent 流程用 Java 代码直写（AgentBase + doExecute）。编排轻量化，不做通用工作流引擎。
- **后续评估：** 如果 Agent 数量到 5+ 且流程复杂度持续增长，再评估引入 LangGraph。

### 2. 调研过 Dify / LangGraph / OpenAI Agents SDK 吗？

**问题：** 方案全篇没有提到任何竞品或开源方案的 POC 结论。

**结论：** M1 不强制迁移到这些方案，但保留评估。当前选择"轻量自研 AgentBase + 代码直写流程"是现阶段最务实的路线，但不排斥未来按需引入。这些方案的能力（durable execution、checkpoint、HITL、流式输出）是我们后续可能需要的，不要因为当前自研而闭门造车。

### 3. AI 发展这么快，做深度自研的时机对吗？

**问题：** LLM 编排领域每季度都有重要演进。锁定自研 = 锁定维护成本 = 与整个生态赛跑。

**结论：** 不正面对抗。编排层能不开源就轻量，但领域 Tool + 财务算法（预测引擎、敞口计算、排程优化）是持久壁垒。护城河不在"怎么调步骤"，在"步骤里算出来的是什么"。

## 二、架构设计

### 4. L1-L5 五层模型——每层承载了什么独立的技术决策？

**问题：** 五层拆开后每层只有一个类。架构分层不是越细越好。

**结论：** 不照搬五层模型。架构压成两层：

- **平台层：** AgentBase + ToolExecutor + LlmGateway + AuditService + RBAC
- **垂直层：** 每个 Agent 一个 Java package，自控流程、自定 prompt、自写领域逻辑

### 5. Skill YAML DSL——谁来维护依赖关系？会不会变成低配编程语言？

**问题：** dependsOn 手写、outputKey 引用、Schema 校验、模板变量——这些东西的编写/调试/维护成本全部由开发承担。M2 规划的"YAML 编辑器 + DAG 预览"说明 DSL 本身不好用。为什么不用 Python/TypeScript 写 Skill？

**结论：** 砍掉 YAML DSL。Agent 的执行步骤直接用 Java 代码写——有 IDE 提示、编译检查、单测覆盖、git diff 可读。不需要再发明一门配置语言。

### 6. "Tool 不互相调用，多 Tool 组合只能通过 Skill YAML"——约束还是枷锁？

**问题：** 条件分支怎么办？预测结果异常 → 触发风险扫描，这个逻辑在 YAML 里写不了（只有 onError: abort|continue）。

**结论：** 约束解除。在 AgentBase.doExecute() 的代码里，Agent 自有判断逻辑：调完 Tool A 看结果，决定是否调 Tool B，直接写 if/else。DSL 做不到的条件分支，代码天然支持。

### 7. RuleEngine 只有一个规则（检查数据非空）——凭什么说是"确定性执行防线"？

**问题：** 文档大量篇幅讲 RuleEngine 做合规校验、阈值拦截、口径验证。代码只有一条规则。新增规则要改 Java 代码 → 编译 → 部署。不叫"业务规则可配置"。

**结论：** 砍掉独立 RuleEngine 模块。业务规则回归各 Agent 自己的校验方法。例如 CashflowAgent 里的 validateLedgerData()。规则就是代码，代码就是规则——不强求"可配置"。

### 8. 数据持久化双轨——4 人团队维护两套存储方案？

**问题：** 数据库 + 本地 md 文件/向量数据库 = 两套 DAO、两套查询、两套运维。哪个场景必须用本地 md 文件？

**结论：** 砍掉本地 md 文件双轨。一期只用 PostgreSQL。需要轻量部署时用 H2 嵌入式数据库，不需要引入向量库。

### 9. Electron 桌面端——财务人员需要吗？

**问题：** 引入版本分发、跨平台兼容、安装包构建——换来了浏览器不能干的事？

**结论：** 一期不做。先浏览器交付。如果后期客户明确要求桌面端，再评估是 Electron 还是 PWA。

## 三、Agent 设计

### 10. IntentClassifier 硬编码关键词规则 vs LLM 路由

**问题：** 当前 3 条硬编码规则。用户说"帮我算一下接下来三个月的钱够不够"就 miss。但 30 个意图写硬编码规则又是噩梦。

**结论：** 用 LLM 做意图分类。两层路由架构：

- **Gatekeeper（fast model）：** 判断"是不是任务"，闲聊走短路径
- **AgentRouter（fast model）：** 匹配到具体 Agent

路由失败 → 引导用户选择 Agent，前端展示快捷选项 chip。

### 11. "LLM 只负责局部叙事"——AI 的价值到底是什么？

**问题：** 如果 AI 的唯一产出是一段 markdown 说明文字，这套系统的核心竞争力跟传统规则引擎有什么区别？

**结论：** 一期 AI 的合理职责：
- 识别用户意图并补齐参数（routing）
- 解释预测结果、总结异常原因
- 生成管理层简报
- 回答用户对预测结果的追问

一期 AI 不承担：
- 自动决定资金调拨/出款
- 自动改变财务口径
- 绕过审批链路
- 生成不可追溯的最终财务结论

LLM 的叙事必须引用结构化预测结果，不允许凭空编造数据。这是"AI 为肉"的边界。

### 12. 预测计算应该放 Tool 层还是 Agent 内部？

**问题：** 当前 compute.stats 注册为一个独立 Tool。但预测计算是领域核心逻辑，应该受 Tool 白名单和权限约束吗？

**结论：** 预测计算不放 Tool 层。改放到 agent 内部——`ForecastEngine` 作为 CashflowAgent 的私有领域类。Tool 层只放"外部感知"类能力：取数据（ledger.fetch）、导出文件（delivery.excel）、调用外部系统。判断标准：这个能力未来会跨 Agent 复用吗？不会 → 不放 Tool 层。

### 13. 随机对话 / 匹配不到 Agent 怎么办？

**问题：** 用户可能说"你好"、闲聊、问"什么是现金流"，或者问一个当前不存在的 Agent 领域（比如"帮我做税务分析"）。

**结论：** 三层兜底：
1. Gatekeeper 拦截闲聊 → 走纯对话通道，不触发任何 Agent
2. AgentRouter 匹配不上 → Agent 回复里带引导："您的问题我暂时无法确定由哪个模块处理，请问是关于[现金流/外汇/排程]的吗？"前端同时展示可选 Agent chip
3. 不属于任何已部署 Agent 的领域 → 记录路由日志，作为后续 Agent 排期的数据输入

### 14. AgentBase 一开始就抽厚还是先跑通再抽象？

**问题：** 只做一个 CashflowAgent 可以用最简单的抽象。但后续会上多个 Agent（外汇、排程），过度抽象和欠抽象都有代价。

**结论：** 一开始就抽厚。AgentBase 提供：
- execute() final 骨架：trace 初始化 → doExecute → 异常兜底 → audit → 返回
- callTool() / callLlm() / audit() —— 自动挂 trace，Agent 开发者不用手动记
- doExecute() abstract —— 子类只关心业务步骤
- routeHint() / model() —— 可选覆盖

Agent 开发者只需要实现：info + routeHint + classify + doExecute + tools。其余平台统管。

## 四、自定义 Agent / DataAgent / 数字人

### 15. 产品要的"选 Tool + 写 Prompt 就能创建 Agent"能做吗？

**问题：** 产品希望管理员勾几个 Tool、写一段 system prompt，就能快速创建一个 Agent 产品。

**结论：** 分场景回答。

**能做的（L2 配置型 Agent）：** 查询型场景——一个 Tool 完成一个意图，LLM 翻译结果。例如：头寸查询、汇率播报、日报导出。这类 Agent 的执行逻辑是通用模板，可以配置生成。**一期不做，二期交付。**

**不能做的（流程型 Agent）：** 任何涉及多步骤串联、复杂计算、业务规则判断的 Agent——现金流预测、外汇敞口、智能排程。原因：
- Tool 之间的调用顺序不是 prompt 能保证的（LLM 会自主决策，不保证确定性）
- 中间数据处理（字段映射、口径转换、清洗聚合）Tool 的 inputSchema 覆盖不了
- 异常判断（数据不全会怎样、极端值怎么处理）需要业务知识，prompt 里写不完
- 市面上没人真正做到——Dify 的工作流能配的人也是技术人员，GPTs 的实际使用极少出现在多步骤工具链场景

### 16. 三层 Agent 创建体系

| 层级 | 能力 | 一期 | 二期 |
|------|------|:----:|:----:|
| L1 参数级 | 修改已有 Agent 的名称/描述/图标/权限/用户分配 | ✅ | 增强 |
| L2 查询型 | 选 Tool + 写 Prompt = 查询型 Agent（单步调用约束） | — | ✅ |
| L3 流程型 | 可视化编排多步骤 + 条件分支 + 自定义规则 | **永不** | **永不** |

L3 永不做的原因：
1. 这是 Skill Studio 翻版，会把团队拖进工作流引擎维护黑洞
2. Dify/LangGraph/Temporal 投入远超我们团队规模
3. 财务领域的编排需要特定会计知识，极少有人能胜任
4. ROI 为负——投入大于产出

### 17. "数字人"的隐喻可能导致产品误判

**问题：** "为每个财务主管配一个专属数字人"听起来先进。但实际价值存疑。

**结论：** 财务人员每天的核心操作是看头寸、做预测、审报表。换专属头像 + 专属 prompt 不解决任何实际问题。L1 确实能做"为审计团队复制一个只读版现金流 Agent"，但"给华南区财务总监配专属数字人"在业务价值上需要论证。建议产品团队先跟真实用户沟通，确认这类需求是否存在。

## 五、产品逻辑

### 18. "13周现金流预测"核心价值在编排还是在预测模型？

**问题：** 如果 compute.stats 返回硬编码数据，编排得再漂亮价值为零。现金流预测的壁垒是模型准确性，不是调用链路确定性。

**结论：** 工程资源优先级重新分配：
- **高优先级：** 预测算法（ForecastEngine）——移动平均 + 周期性因子，口径明确可解释
- **高优先级：** 数据连接（TMS直连）——真实数据输入
- **中优先级：** 对话体验（LLM routing + 对话式交互）
- **低优先级：** 编排灵活性（不需要，流程固定）
- **不做：** 编排平台化（Skill Studio / YAML 编辑器）

### 19. "同一请求两次 Tool 序列完全一致"——这是谁的需求？

**问题：** 用户关心预测准不准、简报有没有用，不在乎 Tool 序列是否一致。这个验收标准是开发者写给开发者看的。

**结论：** 验收标准改为：
- **业务验收：** 财务人员能独立完成预测；结果能追溯数据来源和计算口径；导出的报表能进现有工作流；用户愿意基于结果追问
- **技术验收：** 同一输入数据和口径下，核心数值结果稳定一致（允许 LLM 措辞变化，但关键结论必须受结构化数据约束）；每次执行有完整 trace；权限不足不能执行

### 20. Skill 版本管理、灰度发布、回滚——现在需要吗？

**问题：** 你在做内部工具，不是在发布 SaaS 平台。3 个 Skill 需要灰度？

**结论：** 一期不做。Agent 就是代码，用 git 做版本管理。不需要在运行时搞灰度。等 Agent 数量到 5+ 且有外部客户时再评估。

### 21. 三维权限模型——10 人团队需要吗？

**问题：** 人→资源 × Agent→资源 × 人→Agent 的交叉校验，开发成本远超管理价值。

**结论：** 一期保留现有 RBAC（已实现，质量不错），只做到：
- 用户 → 角色 → 菜单/资源权限（已有）
- Agent 声明所需权限级别，执行时校验（已有）
- 不做：人→Agent 的细粒度绑定、三维交叉界面

- 用户能看懂预测结果和简报
- 记录用户"看不懂/不会用/不信任"的地方

## 八、Skill 定位

### 29. 这个产品需要 Skill 吗？

**问题：** 原方案将 Skill 定义为"可版本化的执行资产 = Playbook YAML + Tool 白名单 + Step 编排 + Rule 校验 + LLM 叙事 + 权限 + Trace/Audit"。回头看，这套东西到底要不要保留？

**分析：** Skill 这个概念只在两个条件下成立：
1. 同一个能力被多个 Agent 需要（复用）
2. 非技术人员需要独立理解/管理这个能力（管理）

当前只有一个 CashflowAgent，两个条件都不满足。

如果等第二个 Agent 上线后，发现两段代码确实有重复，抽一个公共方法或 service——一个下午的事。如果先定义 Skill 的契约格式、注册机制、版本管理、配置界面，然后发现只被一个 Agent 用过一次——几周的沉默成本。

**结论：** 不要 Skill 这个概念层。

| 原方案 Skill 想做 | 实际替代方案 |
|-------------------|-------------|
| 流程编排（step 顺序+依赖） | AgentBase.doExecute() 代码直写 |
| Tool 白名单 | Agent.tools() 返回值 |
| 参数校验 | Agent 内部的 validateXxx() 方法 |
| LLM 叙事 | callLlm(promptKey, vars) |
| 权限校验 | Agent info().permissionLevel() + RBAC |
| Trace/Audit | AgentBase.execute() final 骨架自动挂载 |
| 输出契约 | AgentResult 结构化返回 |

每一项都有更简单的落地方式，不需要引入新概念。

### 30. 不叫 Skill，共享能力怎么管理？

**共享的三样东西：**

| 共享物 | 管理方式 | 当前状态 |
|--------|---------|---------|
| Tool | ToolExecutor + ToolHandler 接口 | 已有，保留 |
| Prompt 模板 | 模板文件或 DB 表（Agent 代码调用） | 先在 Agent 代码里写字符串，复用出现时再抽 |
| 领域计算引擎 | Java Service（如 ForecastEngine） | 先在 Agent 内部作为私有类，被第二个 Agent 需要时升级为共享 Service |

共享能力从代码里自然长出来，不预先设计注册机制或管理后台。

### 31. 竞品都叫 Skill，我们不叫会不会被动？

市场上"Skill"在不同产品里含义完全不同：

| 产品 | 它的 Skill 是什么 | 我们的等价物 |
|------|------------------|-------------|
| Claude Skill | SKILL.md 文件，装上下文+脚本，增强模型知识 | Prompt 模板 + Tool 绑定 |
| Dify Tool | YAML/JSON 描述的 API 调用，注册到工作流 | ToolHandler |
| OpenAI GPTs Action | OpenAPI schema，GPT 可调用 | ToolHandler |

他们叫 Skill 的，我们叫 Tool 或 Prompt 模板。名字不同，能力对等。不需要跟着叫。

### 32. 前端叫 Skill，内部怎么做？

**定位：** "Skill"在前端和对外表达中保留——它是用户能理解的"Agent 会干什么"。但对内阐述清楚：**我们的 Skill 不是 YAML DSL，不是配置表单，是 Agent 代码里的结构化能力声明。**

**对内一句话：**

> Skill = Agent 内部一个可被命名、可被展示、可被路由的**能力单元**。执行逻辑在 Agent.doExecute() 的 Java 代码里，不经过 YAML 或可视化编排。

**对内实现（研发看到的）：**

```java
@Component
public class CashflowAgent extends AgentBase {

    // Skill 声明 = Agent 代码里的结构化元数据
    @Override
    public AgentInfo info() {
        return AgentInfo.builder()
            .id("cashflow")
            .name("现金流预测")
            .skills(List.of(
                SkillInfo.of("forecast", "13周滚动预测", "基于TMS分类账数据生成滚动现金流预测"),
                SkillInfo.of("explain",  "预测结果解释", "AI解释关键变动和风险"),
                SkillInfo.of("export",   "导出Excel报表", "生成标准化预测报表")
            ))
            .tools(List.of("ledger.fetch", "delivery.excel_report"))
            .build();
    }

    // Skill 执行 = doExecute() 里的代码分支
    @Override
    public AgentResult doExecute(Intent intent, SessionContext ctx) {
        return switch (intent.action()) {
            case "forecast" -> doForecast(intent, ctx);  // Java代码，不是YAML
            case "explain"  -> doExplain(intent, ctx);
            case "export"   -> doExport(intent, ctx);
            default         -> doChat(intent, ctx);
        };
    }
}
```

**跟原方案的本质区别：**

| | 原方案 | 现在 |
|------|--------|------|
| Skill 定义在哪 | 独立 YAML 文件 | AgentInfo.skills() 元数据 |
| Skill 怎么执行 | SkillExecutor 解析 YAML steps | Agent.doExecute() 代码直写 |
| Skill 怎么管理 | 管理后台 CRUD + 版本 | 代码 git 版本管理 |
| 新增 Skill 怎么做 | 写 YAML + 注册 Tool + 补 Intent 规则 | 在 doExecute() 里加一个 case 分支 |
| 前端展示什么 | Skill YAML 的结构信息（steps/tool/rule） | Skill 名称+描述（用户能看懂的一句话） |
| 对外讲什么 | "我们有 Skill 编排引擎" | "每个 Agent 内置多个专业能力" |

**总结：** "Skill"这个叫法保留——用户听得懂，售前讲得通。但实现上它是一个 Agent 内部的命名能力单元，不是独立配置文件，不是可视化编排节点，不是可配置资产。**名字借过来，实现走自己的路。**
