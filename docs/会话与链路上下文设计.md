# 会话与链路上下文设计

> 合并重构自 `conversation-context-design.md` + `trace-context-design.md`。
> 本文档覆盖会话记忆、链路追溯、上下文注入、路由感知、超限策略的完整闭环。

---

## 1. 概念定义

| 中文名称 | 英文标识 | 生命周期 | 谁生成 | 说明 |
|----------|----------|----------|--------|------|
| 会话 | `sessionId` | 用户点击"新消息"前保持不变 | **前端** | 同一会话内所有轮次共享上下文 |
| 轮次 | `turnId` | 一次用户发送→AI回复 | **前端** | 一问一答为一个轮次 |
| 链路 | `traceId` | 一次 HTTP 请求 + 内部异步链路 | **后端 TraceIdFilter** | 分布式追踪 ID，日志/SSE/审计共用 |
| 跨度 | `spanId` | 规划中，未实现 | — | 单次 LLM/Tool/DB 调用的子追踪 |

### 归属权规则

```
前端拥有:  sessionId, turnId
后端拥有:  traceId (TraceIdFilter 生成或复用 X-Trace-Id)
```

**为什么 turnId 归前端：** 前端在用户每次发送消息时生成新的 turnId。后端不做 fallback 生成——如果前端未传 turnId，后端拒绝请求（返回 400）。这与 sessionId 的归属模型一致。

**为什么 traceId 归后端：** traceId 是分布式追踪 ID。前端可以传 `X-Trace-Id` 作为建议值（方便端到端关联），但后端必须验证/覆盖，防止客户端伪造导致日志污染。

 │──────────────────────────────────>│                       │                      │                    │                      │
 │                                   │ resolve traceId       │                      │                    │                      │

 │                                   │                       │                      │                    │ ├─ callLlm()        │
 │                                   │                       │                      │                    │ ├─ callTool()       │
 │                                   │                       │                      │                    │ └─ emit events →    │
 │                                   │<── SSE events ──────────────────────────────────────────────────│                      │
 │<── SSE stream ───────────────────│                       │                      │                    │                      │
 │                                   │                       │                      │                    │                      │
 │                                   │                       │ appendTurn() ─────────────────────────────────────────────────>│
 │                                   │                       │                      │                    │                      │
 │                                   │ MDC.clear()           │                      │                    │                      │
```

### MDC 注入节点与清理时序

| 节点 | 写入 | 清理时机 |
|------|------|----------|
| `TraceIdFilter` | traceId, userId, sessionId, turnId | `finally` 块（请求结束） |
| `AgentController` | sessionId, turnId（再次确认） | 请求结束随 TraceIdFilter 清 |
| `AgentBase.execute()` | agentId | `finally` 块（Agent 执行结束） |
| SSE 异步线程 | 通过 `TraceContext.wrap()` 恢复 | 异步任务 `finally` |

**关键约束：** `TraceIdFilter.MDC.clear()` 在 `finally` 中执行。SSE 异步线程在 `filterChain.doFilter()` 返回之前就脱离了请求线程，因此异步线程必须在进入时通过 `TraceContext.wrap()` 显式捕获 MDC。当前 `AgentController.askStream` 第 205 行已正确使用。

## 3. 会话记忆管理

### 3.1 存储结构

`AgentSessionManager` 当前使用 `ConcurrentHashMap<String, CopyOnWriteArrayList<ConversationTurn>>` 内存存储。

```
会话: sess-001
├── turn-001: {userInput, agentReply, agentId, turnId, timestamp}
├── turn-002: {userInput, agentReply, agentId, turnId, timestamp}
└── turn-003: {userInput, agentReply, agentId, turnId, timestamp}
```

**ConversationTurn 结构（修正后）：**

```java
public record ConversationTurn(
    String turnId,      // 轮次 ID，来自前端 X-Turn-Id
    String userInput,   // 用户原始输入
    String agentReply,  // Agent 最终回复
    String agentId,     // 处理此轮次的 Agent
    long timestamp      // epoch millis
) {
    public int charCount() {
        return userInput.length() + agentReply.length();
    }
}
```

**修正说明：** 原设计缺少 `turnId` 字段，导致无法按轮次查询 audit_log、无法定位单轮问题。修正后每个 ConversationTurn 携带 turnId。

### 3.2 上下文预算

| 参数 | 当前值 | 说明 |
|------|--------|------|
| 最大轮次数 | 20 | 超出后从最旧轮次开始丢弃 |
| 最大字符数 | 12,000 | 从最新轮次向前累加，超出部分截断 |
| 警告阈值 | 9,000 | 触发前端上下文警告 |

**trimToBudget 算法（已实现，无需修改）：** 从最新轮次向前累加字符数，直到达到 12,000 字符上限。保证最近的对话优先保留。

### 3.3 上下文超限策略（修正）

原设计只弹警告。修正为**警告 + 自动摘要**两级策略：

| 阶段 | 条件 | 行为 |
|------|------|------|
| 正常 | chars < 9,000 | 完整注入最近 20 轮 |
| 警告 | chars ≥ 9,000 | 前端展示"建议新消息"提示 + 继续注入 |
| 压缩 | chars ≥ 12,000 或 turns ≥ 20 | 将前 50% 的轮次压缩为一条摘要，只保留最近 10 轮原文 |

**摘要格式：**

```
[会话摘要] 此前用户讨论了以下话题：查询深圳天气、创建待办"提交周报"、询问下周三日期。关键信息：用户所在地深圳龙岗区。
```

摘要由 LLM 在后台异步生成（不阻塞当前请求），存入 `sessionSummary` 字段。下一轮对话时，摘要作为 system message 注入到 LLM 上下文。

### 3.4 会话生命周期

```
创建: 首次 POST /api/agent/ask 携带新 sessionId
活跃: 持续 POST 同一 sessionId
闲置: 30 分钟无新请求
过期: 闲置超时后自动清理（内存 + Redis）
销毁: 用户点击"新消息" → POST /api/agent/session/clear
```

**需要新增的端点：**

```
POST /api/agent/session/clear
Body: { "sessionId": "sess-001" }

行为:
1. AgentSessionManager.clearConversation(sessionId)  → 清理内存轮次
2. 取消该 session 所有 pending 确认
3. 关闭该 session 的活跃 SSE 流
4. 返回 { "cleared": true }
```

### 3.5 会话淘汰

当前实现缺失，需要补充：

- **内存淘汰：** 后台定时任务每 5 分钟扫描，清理超过 30 分钟未访问的会话
- **Redis 淘汰：** key 设置 TTL 30 分钟，每次访问刷新 TTL
- **优雅降级：** Redis 不可用时回退到纯内存模式

## 4. 上下文注入 Agent

### 4.1 注入位置

`ReActAgentBase.doExecute()` 在构建 LLM 消息列表时，调用 `appendConversationHistory()`：

```java
// 消息顺序
1. System: systemPromptBase + 工具列表 + ReAct 格式说明
2. System: "以下是当前会话的最近上下文..."  (仅当有历史轮次时)
3. User: 历史轮次1的userInput
4. Assistant: 历史轮次1的agentReply
5. User: 历史轮次2的userInput
6. Assistant: 历史轮次2的agentReply
   ...
N. User: 当前轮次的 userInput
```

### 4.2 注入规则

- 只注入最近 20 轮或 12,000 字符以内的内容
- 如果存在 `sessionSummary`，在历史轮次前插入一条 system 消息作为摘要
- 上下文指令明确告诉 LLM："用户没有点击新消息时，后续输入默认延续这个上下文"
- 历史轮次不包含 tool/confirm 交互细节，只包含最终 userInput 和 agentReply

## 5. 路由与上下文

### 5.1 两层路由信号

AgentRouter 使用两层信号判断目标 Agent：

**第一层：当前消息关键词（确定性）**

```java
text.contains("天气") → life-assistant
text.contains("现金流") → cashflow
// ... 更多关键词
```

**第二层：上下文感知（追问识别）**

```java
// 如果当前消息短小/指代性强，且近期上下文提到某领域关键词
(contextText.contains("天气") && looksLikeFollowUp(text)) → life-assistant
(contextText.contains("待办") && looksLikeFollowUp(text)) → life-assistant
```

`looksLikeFollowUp` 判定条件：
- 消息长度 ≤ 30 字符
- 或包含指代词：这个、那里、刚才、上面、继续、深圳、龙岗（地点补充）

### 5.2 上下文窗口

路由只检查最近 **3 轮**上下文文本（`recentConversationText` 取最近 3 轮拼接）。不需要检查全部 20 轮——最近几轮足以判断追问意图。

## 6. 链路追踪

### 6.1 HTTP 头部约定

| 头部 | 方向 | 必填 | 说明 |
|------|------|:---:|------|
| `X-Trace-Id` | 请求 | 否 | 前端可传建议值，后端总是覆盖 |
| `X-Session-Id` | 请求 | 是 | 会话 ID |
| `X-Turn-Id` | 请求 | 是 | 轮次 ID，前端每次发送时新建 |
| `X-Trace-Id` | 响应 | — | 后端确定的 traceId |
| `X-Session-Id` | 响应 | — | 回显 sessionId |
| `X-Turn-Id` | 响应 | — | 回显 turnId |

### 6.2 日志格式

```
2026-05-03 14:30:01.234 INFO [agent] trace=t-abc123 session=sess-001 turn=turn-003 user=zhangsan agent=life-assistant - react llm response: iteration=0, contentLen=320
```

所有关键日志节点必须包含 trace + session + turn + user + agent（agent 仅在 Agent 执行阶段有值）。

### 6.3 查询模式

| 排查目标 | 查询方式 |
|----------|----------|
| 定位单次失败请求 | 按 `traceId` 搜索 |
| 回溯整段对话上下文问题 | 按 `sessionId` 搜索，按时间排序 |
| 定位某一轮用户消息 | 按 `turnId` 搜索 |
| 审计某用户全部操作 | 按 `userId` + 时间范围搜索 |

## 8. 并发安全

### 8.1 MDC 线程安全

- MDC 是 `ThreadLocal`，请求线程自动隔离
- 异步线程池（`streamExecutor`）通过 `TraceContext.wrap()` 传递 MDC 上下文
- `AgentBase.executionState` 使用 `ThreadLocalExecutionState`，确保单例 Agent 的并发请求互不干扰

### 8.2 SSE 流互斥

- 每个 `sessionId` 同时只保留一个活跃 SSE 流
- 新流创建时，旧流被 `completeEmitter` 关闭，防止交错写入
- 多节点部署时，此逻辑需要迁移到 Redis（`SETNX` + TTL）

### 8.3 Agent 单例安全

- Agent 是 Spring 单例 Bean
- 请求级可变状态放在 `executionState`（ThreadLocal Map）
- 禁止在 Agent 子类中定义实例字段存储请求状态

## 9. 端到端场景走查

### 场景一：新会话

```
1. 用户打开工作台 → 前端生成 sessionId=sess-001
2. 用户输入"今天天气怎么样？" → 前端生成 turnId=turn-001
3. POST /api/agent/ask/stream {sessionId: sess-001, turnId: turn-001, input: "..."}
4. TraceIdFilter: MDC → {traceId: t-001, sessionId: sess-001, turnId: turn-001, userId: zhangsan}
5. AgentRouter: keyword match → life-assistant
6. LifeAssistantAgent.doExecute():
   - snapshot(sess-001) → 空（首轮）
   - callLlm → weather.lookup tool → FINAL_ANSWER
7. sessionManager.appendTurn(sess-001, "今天天气怎么样？", "深圳今天晴...", "life-assistant")
8. 返回 SSE 事件流
```

### 场景二：追问（上下文延续）

```
1. 用户在同一会话输入"深圳龙岗区" → turnId=turn-002
2. snapshot(sess-001) → [turn-001: "今天天气怎么样？" / "请问您想查询哪个城市..."]
3. appendConversationHistory 注入 turn-001 的 user/assistant 消息
4. LLM 看到上下文：上一轮用户问天气 → 本轮用户说"深圳龙岗区" → 理解为补充地点
5. AgentRouter: keywordMatch 检查近 3 轮上下文 → 包含"天气" + 当前消息是追问 → life-assistant
6. LLM 调用 weather.lookup(location="深圳龙岗区")
7. sessionManager.appendTurn(sess-001, "深圳龙岗区", "深圳龙岗区今天晴...", "life-assistant")
```

### 场景三：上下文超限 → 自动摘要

```
1. sess-001 已有 19 轮，字符数累计 11,800
2. 用户发送第 20 轮 → snapshot 返回 nearLimit=true
3. AgentController 发射 context warning 事件："建议点击新消息"
4. doExecute 正常执行，第 21 轮时将最旧 10 轮发送给 LLM 做摘要
5. 摘要存入 sessionSummary："此前用户讨论了天气查询、待办创建、日期询问..."
6. 后续轮次：摘要作为 system 消息注入，原文只保留最近 10 轮
```

### 场景四：点击"新消息"

```
1. 用户点击"+ 新消息"按钮
2. 前端: POST /api/agent/session/clear {sessionId: sess-001}
3. 后端: clearConversation(sess-001) → 移除内存中的轮次 + 取消 pending 确认 + 关闭 SSE
4. 前端: 生成新 sessionId=sess-002，清空消息列表
5. 后续请求使用 sess-002，从空白上下文开始
```
